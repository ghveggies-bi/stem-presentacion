Slide 2

Why SharePoint is not a databasePor qué SharePoint no es una base de datos

SharePoint stores files very well. But using it as a data engine for leadership reports is building on sand: it works until a token expires, a header changes, or a file grows too large. SharePoint guarda archivos muy bien. Pero usarlo como motor de datos para reportes de dirección es construir sobre arena: funciona hasta que un token caduca, un encabezado cambia, o el archivo crece de más.

Refresh stabilityEstabilidad de actualización

Today a report stops updating because a SharePoint access token expired — a security credential that Microsoft 365 silently rotates — or because the web connector dropped. Nobody notices until the meeting. A SharePoint token is a temporary key that the platform reissues on its own schedule; when it changes, the report is locked out and the data goes stale without any error in sight. The Data Warehouse removes that fragility: loading becomes a controlled, monitored process with an immediate alert if anything fails, before it ever reaches a dashboard. Updating stops depending on the luck of the session. Hoy un reporte deja de actualizar porque un token de acceso de SharePoint caducó —una credencial de seguridad que Microsoft 365 renueva en silencio— o porque el conector web se cayó. Nadie lo nota hasta la junta. Un token de SharePoint es una llave temporal que la plataforma reemite por su cuenta; cuando cambia, el reporte queda bloqueado y el dato se queda viejo sin que aparezca ningún error. El Data Warehouse elimina esa fragilidad: la carga es un proceso controlado y monitoreado, con aviso inmediato si algo falla, antes de que llegue a un tablero. La actualización deja de depender de la suerte de la sesión.

Governance and single source of truthGobernanza y fuente única de verdad

Today the business logic — how each KPI is calculated — lives locked inside each individual .pbix file. If that person leaves or the file is lost, the definition of the number is lost with it. In the Data Warehouse the rules live in one central, auditable, backed-up place. Every report reads the same truth, not each file's own interpretation. Hoy la lógica del negocio —cómo se calcula cada KPI— vive secuestrada dentro de cada archivo .pbix individual. Si esa persona se va o el archivo se pierde, se pierde con él la definición del número. En el Data Warehouse las reglas viven en un lugar central, auditable y respaldado. Todos los reportes leen la misma verdad, no la interpretación de cada archivo.

Decoupling the visual layerDesacoplamiento de la visualización

Today if someone changes a column in a source Excel, every report that depended on it breaks in a chain, and the team spends days repairing. With the Data Warehouse in the middle, that change is absorbed and fixed once, in SQL. The visual layer never even notices. One repair point instead of ten. Hoy si alguien cambia una columna en un Excel de origen, se rompen en cadena todos los reportes que dependían de él, y el equipo pasa días reparando. Con el Data Warehouse en medio, ese cambio se absorbe y se corrige una sola vez, en SQL. La capa de visualización ni se entera. Un punto de reparación en lugar de diez.

Historical scalabilityEscalabilidad histórica

Power BI chokes as history grows: memory is limited and not expandable. A SQL engine is built precisely to index and query large historical volumes without degrading. We can keep years of operations and query them in seconds, without the dashboard becoming slow or running out of memory. The system grows with the business, not against it. Power BI se asfixia cuando el histórico crece: la memoria es limitada y no expandible. Un motor SQL está hecho precisamente para indexar y consultar grandes volúmenes históricos sin degradarse. Podemos guardar años de operación y consultarlos en segundos, sin que el tablero se vuelva lento ni se quede sin memoria. El sistema crece con el negocio, no contra él.

To run this migration in production, we require the following actions coordinated strictly with the data infrastructure team. These are not business asks — they are technical enablement only your team can grant. Para ejecutar esta migración en producción, requerimos las siguientes acciones coordinadas estrictamente con el equipo de infraestructura de datos. No son peticiones de negocio: son habilitaciones técnicas que solo su equipo puede otorgar.

1

Storage and schema on GHS_BI serverAlmacenamiento y esquema en el servidor GHS_BI

Confirm dedicated storage on the on-premise SQL Server 2022 host (192.168.**.54, where the GHS ERP read database GHS_BI already resides) and authorize creating the main schema under the name DWH_GHVeggies. This consolidates the repository in the same ecosystem where the operational transactional data already lives, minimizing latency and avoiding additional cloud licensing. Confirmar almacenamiento dedicado en el host SQL Server 2022 on-premise (192.168.**.54, donde ya reside la base de lectura del ERP GHS, GHS_BI) y autorizar la creación del esquema principal bajo el nombre DWH_GHVeggies. Esto consolida el repositorio en el mismo ecosistema donde ya vive el dato transaccional operativo, minimizando latencia y evitando licencias adicionales de nube.

2

System identities for M365 accessIdentidades de sistema para acceso a M365

CRITICALCRÍTICO

This is the point that causes the most outages and demands the most firmness. We need two permanent system identities, both exempt from password rotation. First, a Service Account in Microsoft Entra ID for cross-tenant pipeline authentication: relying on a personal user is the main cause of Dataflow failures today, because every M365 password reset blocks the entire analytical pipeline corporate-wide. Second, an App Registration in Microsoft Entra ID (with a client secret or certificate) granted the application permission Sites.Read.All on Microsoft Graph, with admin consent on both SharePoint tenants — GHVeggies and Ganfer. Without these, the system will keep failing intermittently no matter how well the warehouse is built. This is not a technical luxury; it is the difference between a reliable system and one that breaks every few weeks. Este es el punto que más caídas causa y el que más firmeza requiere. Necesitamos dos identidades de sistema permanentes, ambas exentas de rotación de contraseñas. Primero, una Service Account en Microsoft Entra ID para la autenticación de la tubería entre tenants: depender de un usuario personal es hoy la causa principal de fallos en los Dataflows, porque cada reseteo de contraseña de M365 bloquea toda la tubería analítica a nivel corporativo. Segundo, un App Registration en Microsoft Entra ID (con client secret o certificado) con el permiso de aplicación Sites.Read.All sobre Microsoft Graph, con admin consent en los dos tenants de SharePoint —GHVeggies y Ganfer. Sin esto, el sistema seguirá fallando de forma intermitente sin importar qué tan bien esté construido el Data Warehouse. No es un lujo técnico; es la diferencia entre un sistema confiable y uno que se cae cada pocas semanas.

3

Execution permissions on DWH_GHVeggiesPermisos de ejecución sobre DWH_GHVeggies

Grant read and write permissions on the DWH_GHVeggies schema for the orchestration tooling (Python and scheduled routines), so the automated load processes can run without permission bottlenecks during development. We will provide the exact minimum access level required; what we need from your side is the authorization to operate within that governed schema on a defined schedule. Otorgar permisos de lectura y escritura sobre el esquema DWH_GHVeggies para las herramientas de orquestación (Python y rutinas programadas), de modo que los procesos de carga automatizada corran sin cuellos de botella por permisos durante el desarrollo. Nosotros entregamos el nivel mínimo de acceso exacto requerido; lo que necesitamos de su parte es la autorización para operar dentro de ese esquema gobernado en un horario definido.

4

Joint certification window (Shadow Mode)Ventana de certificación conjunta (Shadow Mode)

We will not switch off the legacy architecture until the new Data Warehouse tables match the current reports at 100%. We need to agree on a transition and co-validation window with the Data Analytics team. The go-live must run in parallel — Shadow Mode — and be signed off by both teams, guaranteeing zero disruption to the company's KPIs. No apagaremos la arquitectura legacy hasta que las nuevas tablas del Data Warehouse coincidan al 100% con los reportes actuales. Necesitamos acordar una ventana de transición y co-validación con el equipo de Data Analytics. La liberación a producción debe correr en paralelo —Shadow Mode— y ser firmada por ambos equipos, garantizando cero disrupción en los KPIs de la empresa.
